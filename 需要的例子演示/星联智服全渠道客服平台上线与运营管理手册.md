# 星联智服全渠道客服平台上线与运营管理手册

> 文档密级：内部使用  
> 文档编号：CS-OPS-PLAT-2026-007  
> 发布部门：客户服务中心 / 研发中心 / SRE 团队 / 数据平台部  
> 会签部门：信息安全部 / 法务合规部 / 财务部 / 培训与赋能部  
> 当前版本：V2.4  
> 适用对象：客服主管、项目经理、交付实施、知识运营、SRE、值班研发、合规审计  
> 关联系统：`StarDesk`、`NovaRAG`、`OnePass`、`KeyFlow`、`BlueMonitor`、`LinkChat`、`DataCleanRoom`、`AuditTrail`

---

## 一、编制目的

本手册用于规范星联智服全渠道客服平台在“需求评估、项目立项、知识接入、机器人配置、灰度上线、生产发布、值班运营、质量复盘、数据治理、异常应急”全链路中的职责边界、操作规则和协同机制，确保不同团队在执行过程中遵循统一口径，降低以下常见风险：

1. 平台功能已经上线，但客户业务规则未完成确认，导致配置逻辑和真实流程不一致。
2. 知识库内容已导入，但 RAG 检索链路未经过质量验收，导致机器人命中率低、回答口径漂移或引用错误。
3. 灰度范围和生产范围界定不清，出现“灰度配置误入生产”“生产租户误接测试知识库”的严重事故。
4. 客服主管、实施顾问、知识运营和研发团队各自保留一套说法，导致客户沟通口径混乱。
5. 机器人回答质量下降后，没有统一的指标判定标准和升级路径，只靠经验排查，造成恢复时间过长。

> 核心原则  
> 平台上线不是“功能能跑起来”就结束，而是要同时满足“角色清楚、配置可追溯、知识可验证、故障可恢复、审计有留痕”五项条件。

---

## 二、适用范围

本手册适用于以下场景：

- 新客户首次部署星联智服全渠道客服平台。
- 老客户新增渠道接入，例如从网页客服扩展到企业微信、APP 内嵌、小程序、呼叫中心转人工。
- 客户知识库从 FAQ 升级为结构化文档检索和 RAG 问答。
- 生产环境机器人策略、知识召回参数、人工转接逻辑、服务时间规则发生重要变更。
- 平台重大版本升级，包括但不限于 `NovaRAG` 检索策略升级、对话编排升级、统一认证升级。

以下场景不完全适用本手册，但可参考其中的部分章节：

- 单纯的演示环境搭建。
- 培训环境或售前样例租户。
- 不接入生产知识库的纯测试租户。

## 三、术语说明

### 3.1 平台

本手册中的“平台”默认指星联智服全渠道客服平台的整体能力集合，包含会话接入、用户身份识别、意图识别、知识检索、机器人回答、人工转接、工单协同、监控告警和日志审计等能力。

### 3.2 渠道

渠道是指最终承载客户会话入口的接触点，包括：

- 官网网页客服
- 企业微信客服
- 微信小程序
- APP 内嵌客服
- 公众号消息接入
- 语音机器人和呼叫中心辅助台席

### 3.3 知识域

知识域是指面向某一业务主题独立维护的一组知识资产，通常按产品线、业务流程、客户类型或监管范围划分。例如：

- 产品基础问答域
- 计费与发票域
- 实施上线操作域
- 售后故障处理域
- 合规与政策说明域

### 3.4 父块 / 子块

在平台的文档知识接入过程中：

- 父块：用于回答阶段承载更完整上下文的大块文本。
- 子块：用于召回阶段做精确检索的小块文本。

父块和子块共同组成 Parent-Child 文档结构，平台通过“先命中子块，再回溯父块”的方式兼顾检索精度和回答完整度。

### 3.5 灰度租户

灰度租户是指在正式生产上线前，先接收新版策略、配置或知识库变更的一小部分内部试点或客户试点租户。灰度租户默认不超过生产总量的 **10%**。

## 四、角色与职责矩阵

### 4.1 角色定义

| 角色 | 所属部门 | 核心职责 | 是否有生产权限 |
|------|----------|----------|----------------|
| 项目经理 | 客户成功中心 | 统筹项目节奏、里程碑和跨部门协调 | 否 |
| 实施顾问 | 交付实施部 | 配置业务流程、渠道接入、客户验收培训 | 限定范围 |
| 知识运营专员 | 知识运营部 | 知识整理、问法扩写、召回评估、版本发布 | 否 |
| 机器人策略工程师 | 研发中心 | 对话策略、路由逻辑、RAG 编排参数维护 | 受控 |
| 平台 SRE | 运维部 | 发布执行、容量管理、监控、应急响应 | 是 |
| 安全与合规专员 | 信息安全部 / 法务合规部 | 数据分级、权限审计、合规复核 | 审批权限 |
| 客服主管 | 客户服务中心 | 口径确认、服务时间规则、人工接入规则确认 | 否 |
| 值班研发 | 研发中心 | 复杂故障定位、代码修复、回滚支持 | 受控 |

### 4.2 RACI 责任矩阵

| 事项 | 项目经理 | 实施顾问 | 知识运营 | 策略工程师 | SRE | 合规 | 客服主管 |
|------|----------|----------|----------|------------|-----|------|----------|
| 项目立项 | A | C | I | I | I | I | C |
| 渠道接入准备 | C | A | I | C | C | I | C |
| 知识库初版整理 | I | C | A | C | I | I | C |
| 父子块策略确认 | I | C | C | A | I | I | I |
| 灰度发布 | I | I | C | C | A | I | I |
| 生产变更审批 | I | I | I | C | A | C | I |
| 回答质量验收 | I | C | A | C | I | I | A |
| 故障应急响应 | I | I | I | C | A | I | C |

说明：

- `A`：最终负责
- `R`：执行负责
- `C`：协作咨询
- `I`：知会

### 4.3 职责边界补充

1. 实施顾问可以配置租户级业务参数，但不得直接修改生产环境平台公共策略。
2. 知识运营负责知识内容质量，不直接负责生产发布动作。
3. SRE 负责生产发布与回滚，但不代替业务方做口径确认。
4. 机器人策略工程师负责“为什么这么答”，但不代替知识运营维护原始业务知识。

## 五、项目上线总流程

### 5.1 里程碑概览

完整上线流程划分为 8 个阶段：

1. 需求澄清与立项
2. 渠道与账号准备
3. 知识采集与治理
4. 机器人策略设计
5. 联调与灰度验证
6. 生产发布
7. 上线观察与值班
8. 复盘与持续优化

### 5.2 阶段出口条件

| 阶段 | 阶段出口条件 | 必备材料 |
|------|--------------|----------|
| 需求澄清 | 客户服务场景边界明确 | 需求确认表、角色清单 |
| 渠道准备 | 渠道凭证、账号、白名单全部到位 | 渠道接入检查表 |
| 知识治理 | 知识结构、版本、责任人明确 | 知识清单、知识域映射表 |
| 策略设计 | 问答策略、转人工规则、服务时间策略定稿 | 策略确认单 |
| 灰度验证 | 关键指标达到门槛 | 灰度报告、问题闭环清单 |
| 生产发布 | 审批通过、发布窗口锁定 | 发布单、回滚预案 |
| 上线观察 | 指标稳定、客户反馈可控 | 观察日报 |
| 复盘优化 | 问题归因、行动项闭环 | 复盘报告 |

> 常见误区  
> 很多项目把“客户说可以上线”误判为“已具备发布条件”。  
> 实际上至少还要补齐发布单、回滚预案、灰度结论和权限检查结果。

## 六、需求澄清阶段

### 6.1 必须澄清的业务问题

在项目正式进入配置阶段之前，项目经理和实施顾问必须完成以下问题澄清：

- 客户当前有哪些服务渠道已经在用，哪些渠道本期接入，哪些渠道仅保留后续扩展。
- 客户希望机器人处理哪些问题，哪些问题必须强制转人工。
- 客户的服务时间规则是否存在工作日、节假日、区域和 VIP 客户差异。
- 是否需要多语言、多品牌、多产品线问答隔离。
- 是否存在监管要求，例如留痕、敏感词拦截、人工复核、会话归档时限等。

### 6.2 需求确认模板字段

需求确认单 `FORM-CS-PROJ-11` 至少应包含以下字段：

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| 客户名称 | 是 | 与合同主体一致 |
| 项目编号 | 是 | 统一项目编码 |
| 渠道范围 | 是 | 本期接入渠道清单 |
| 业务场景 | 是 | 售前、售后、工单、催缴等 |
| 服务时间规则 | 是 | 工作日/节假日/特殊日期 |
| 人工转接条件 | 是 | 低置信度、投诉、敏感词等 |
| 数据合规要求 | 是 | 敏感字段、留痕、脱敏要求 |
| 关键联系人 | 是 | 客户侧负责人 |
| 验收标准 | 是 | 指标和里程碑 |

### 6.3 需求变更控制

若需求确认后新增以下变更，必须重新走变更审批：

1. 新增生产渠道接入。
2. 新增高风险业务口径，例如退款、投诉、政策解释。
3. 新增敏感数据处理要求。
4. 调整人工转接原则或服务时间规则。
5. 更换知识源范围，例如从静态 FAQ 改为接入内部制度文档。

## 七、渠道接入与账号准备

### 7.1 渠道接入前置清单

| 检查项 | 负责人 | 完成标准 |
|--------|--------|----------|
| 回调地址白名单 | 实施顾问 | 已在客户侧平台配置 |
| API Key / Secret | 客户联系人 | 已在 `KeyFlow` 密钥库登记 |
| 测试账号 | 客户联系人 | 至少提供 2 个可用测试账号 |
| 机器人头像与欢迎语 | 客服主管 | 已确认最终版本 |
| 工单系统对接字段 | 实施顾问 | 字段映射表完整 |

### 7.2 渠道命名规范

渠道命名统一遵循：

`客户简称-业务域-渠道类型-环境`

示例：

- `HYJT-售后咨询-WEB-PROD`
- `HYJT-售后咨询-WXWORK-GRAY`
- `HYJT-账单催缴-APP-PROD`

### 7.3 密钥管理要求

1. 所有密钥统一保存在 `KeyFlow`，禁止通过聊天工具发送完整明文密钥。
2. 生产密钥与测试密钥必须隔离存储。
3. 任一渠道密钥泄露后，必须在 **2 小时内** 完成更换并补记审计单。

## 八、知识采集与知识治理

### 8.1 知识来源分类

平台支持的知识来源包括但不限于：

- FAQ 文档
- 产品说明书
- 安装部署手册
- 客户成功标准口径
- 政策与制度文件
- 典型工单案例
- 培训问答记录

### 8.2 知识接入前检查

| 检查维度 | 核心问题 | 处理要求 |
|----------|----------|----------|
| 时效性 | 内容是否过期 | 过期内容不得接入生产 |
| 权威性 | 是否来自正式部门 | 口头资料不得直接入库 |
| 粒度 | 是否一段塞了多个主题 | 必要时拆分后再入库 |
| 敏感性 | 是否包含手机号、身份证、账号信息 | 敏感内容需脱敏 |
| 版本归属 | 是否能定位责任人和版本号 | 无版本信息不得上线 |

### 8.3 知识域划分建议

建议至少划分以下知识域：

#### 8.3.1 产品基础问答域

适用于产品是什么、有哪些能力、如何开通等基础问答。

#### 8.3.2 操作与流程域

适用于“怎么做”“在哪设置”“如何审批”“怎么回滚”等操作型问题。

#### 8.3.3 故障处理域

适用于报错排查、接口异常、检索失败、登录问题等技术类问题。

#### 8.3.4 政策与合规域

适用于发票、隐私、保留期限、权限申请、审计流程等规则型问题。

### 8.4 不适合接入机器人知识库的内容

- 未经确认的会议纪要
- 只用于内部讨论的草案
- 依赖实时判断、且无法沉淀稳定规则的客服自由发挥话术
- 含完整客户敏感数据的原始工单明细

## 九、Parent-Child 知识切块配置原则

### 9.1 总体原则

对结构清晰、层级明显、篇幅较长的业务文档，默认采用 Parent-Child 切块结构：

- 父块负责回答阶段的完整上下文
- 子块负责召回阶段的精确命中

### 9.2 推荐配置

默认推荐如下：

| 流水线 | 推荐步骤 | 适用原因 |
|--------|----------|----------|
| 父块流水线 | 结构切块 | 保留章节、条款、流程大单元 |
| 子块流水线 | 语义分块 -> 递归分块 | 优化检索边界并控制长度 |

### 9.3 典型问题举例

#### 9.3.1 为什么不直接让大模型读子块回答

如果直接用子块回答，模型常常只能看到局部片段，容易出现：

- 只答步骤的一部分
- 答案缺少前提条件
- 章节引用不完整
- 对例外场景解释不全

#### 9.3.2 为什么不直接把父块拿去做召回

如果只用父块召回，会导致：

- 检索块过大
- 相似度分数不够敏感
- 命中精度下降
- 多主题混在一起时更容易误召回

### 9.4 什么时候需要双边都走完整流水线

对于章节很多、章节内部还有明显小节、且每个小节本身都很长的制度型文档，可以考虑：

- 父块：结构 -> 语义 -> 递归
- 子块：结构 -> 语义 -> 递归

但必须确认：

1. 父块最终仍然明显大于子块。
2. 父块与子块在长度目标上有差异。
3. 检索和回答的边界仍然可以解释清楚。

### 9.5 参数建议

| 参数 | 父块建议值 | 子块建议值 |
|------|------------|------------|
| 最大字符数 | 1800 到 2200 | 450 到 900 |
| overlap | 120 到 180 | 40 到 120 |
| 语义最小长度 | 400 到 600 | 200 到 300 |
| 语义最大长度 | 1200 到 1600 | 600 到 800 |

## 十、机器人策略设计

### 10.1 策略层次

机器人策略一般分为 4 层：

1. 渠道与租户路由层
2. 业务意图识别层
3. 知识检索与回答生成层
4. 人工转接与补救层

### 10.2 必配策略项

以下策略项在生产上线前必须完成确认：

- 欢迎语与默认兜底语
- 服务时间外回复
- 低置信度转人工阈值
- 敏感词触发规则
- 投诉、退款、取消、法律风险类问题强制转人工
- RAG 无证据兜底文案

### 10.3 不建议交给机器人直接回答的主题

- 退款承诺
- 法律结论
- 赔偿标准
- 合同例外解释
- 医疗、金融、政务等监管行业客户的高风险政策口径

## 十一、灰度验证方案

### 11.1 灰度范围

默认灰度范围不得超过：

- 2 个试点租户
- 10% 会话量
- 1 个工作周

### 11.2 灰度期必看指标

| 指标 | 说明 | 推荐阈值 |
|------|------|----------|
| 首次命中率 | 首轮检索是否命中有效证据 | 不低于 70% |
| 人工转接率 | 转人工占比是否异常升高 | 不高于基线 + 5% |
| 无证据回复率 | RAG 未命中证据比例 | 不高于 15% |
| 用户追问率 | 同一问题多轮追问比例 | 持平或下降 |
| 错误口径工单数 | 因机器人答错产生的工单数 | 趋近于 0 |

### 11.3 灰度期间禁止事项

- 未经审批扩大灰度租户范围
- 同时引入多个高风险变量，例如“模型切换 + 检索参数调整 + 渠道接入”
- 用灰度环境日志替代生产验收结论

## 十二、生产发布流程

### 12.1 发布前检查单

发布前必须逐项确认以下检查单：

1. 发布单是否审批通过。
2. 回滚方案是否明确到人、到步骤、到时间目标。
3. 当前生效知识版本是否与验收版本一致。
4. 父子块切块策略是否已在目标文档上验证。
5. 生产密钥、灰度密钥、测试密钥是否未混用。
6. 监控项和告警项是否已启用。

### 12.2 标准发布步骤

1. 在 `KeyFlow` 锁定发布窗口。
2. SRE 在 `LinkChat` 发布群发出发布开始通知。
3. 发布配置、知识版本、策略版本。
4. 对试点租户先做冒烟验证。
5. 验证通过后逐步放量。
6. 发布结束后记录发布结果和观察项。

### 12.3 回滚触发条件

符合以下任一条件时，默认触发回滚评估：

- 机器人回答错误率持续 15 分钟高于灰度基线两倍。
- `NovaRAG` 无证据回复率高于 25%。
- 人工转接率明显飙升，且客户反馈变差。
- 生产告警连续触发且无法在 30 分钟内稳定。

## 十四、典型故障场景与处理

### 14.1 场景一：检索命中率突然下降

#### 14.1.1 现象

- 用户问法未明显变化
- 机器人开始频繁回复“当前没有足够证据”
- 日志中 `finalTopK` 下降明显

#### 14.1.2 可能原因

1. 新知识版本切块异常。
2. 父子块策略配置错误，导致子块过粗或过碎。
3. 向量索引构建不完整。
4. 检索过滤条件误收紧。

#### 14.1.3 处理步骤

1. 先确认最近一次构建任务是否成功。
2. 对比新旧知识版本的 chunk 数、parent 数、child 数。
3. 抽样检查热门问题在检索链路中的命中证据。
4. 必要时回滚到上一个知识版本。

### 14.2 场景二：回答口径不完整

#### 14.2.1 现象

- 模型给出的内容只有局部步骤，没有补充前提条件
- 同一个问题回答经常漏掉例外条款

#### 14.2.2 可能原因

- 子块命中正常，但父块粒度过小
- 父块没有保留完整章节上下文
- Prompt 中证据预算被过早截断

### 14.3 场景三：人工转接率异常升高

#### 14.3.1 检查顺序

1. 看是不是服务时间策略误配置。
2. 看是否低置信度阈值调得过高。
3. 看最近是否发布了新的知识版本或新策略版本。
4. 看是否存在渠道侧异常，例如用户身份字段缺失。

## 十六、运营质量评估

### 16.1 质量指标分层

质量评估按三层进行：

- 检索层指标
- 回答层指标
- 业务层指标

### 16.2 指标定义

| 指标层 | 指标名称 | 说明 |
|--------|----------|------|
| 检索层 | 有效证据命中率 | 是否召回到真实相关证据 |
| 检索层 | 父块回溯成功率 | 命中子块后是否成功提升为父块 |
| 回答层 | 引用完整率 | 回答是否带出足够引用 |
| 回答层 | 口径一致率 | 是否符合业务标准答案 |
| 业务层 | 首次解决率 | 用户是否在首轮得到可执行答案 |
| 业务层 | 人工接管满意度 | 转人工后客户是否接受 |

### 16.3 评审节奏

- 每周：运营例会复盘
- 每月：版本质量评审
- 每季度：知识域与策略域联合治理评审

## 十七、FAQ

### 17.1 为什么同一个问题线上和测试环境答案不一致？

常见原因有：

1. 两边知识版本不同。
2. 两边父子块策略不同。
3. 测试环境没有接真实渠道上下文。
4. 测试环境使用的是不同模型或不同提示词版本。

### 17.2 为什么知识明明在文档里，机器人还是说没有证据？

需要从以下顺序排查：

1. 文档是否真的被索引成功。
2. 子块切得是否合适。
3. 页码/章节过滤是否误伤。
4. 问题改写是否把查询带偏。
5. 父块回溯后是否被预算截断。

### 17.3 为什么不能把所有制度文件一股脑全接进一个知识域？

因为会造成：

- 召回范围过大
- 主题相近内容互相干扰
- 高风险制度与低风险 FAQ 混答
- 运维定位困难
